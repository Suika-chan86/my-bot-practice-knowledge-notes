# WSL + Docker 运维好习惯

## 1. 数据安全永远是第一优先级

- **永远**把任何会产生数据的容器，用 `-v` 映射到宿主机目录。
- 在折腾任何可能破坏环境的操作前，**先备份**配置文件。
- 重要的对话、记忆、知识库，定期拷贝到另一块硬盘或云端。

## 2. 容器即“牲口”，不是“宠物”

- 不要害怕删容器、重建容器。**只要数据在宿主机，就能秒还原**。
- 用 `docker run` 命令保存成脚本，而不是依赖图形界面记忆参数。
- 使用 `--restart=unless-stopped` 保证容器开机自启，避免手动干预。

## 3. 配置管理要干净

- 一个配置文件只干一件事：人格用“人格管理”，知识库用“知识库”，插件配置各自独立。
- **不要到处散落相同的提示词**（人格里写一套，插件里又注入另一套）。
- 修改配置后，立即在 QQ 发送 `/reset` 清空旧上下文，避免新旧规则打架。

## 4. 网络排查要有逻辑

- 从 `docker ps` 确认容器存活 → 用 `docker logs` 查错误 → 用 `docker exec` 进入容器测试连接。
- 两个容器不能用名字互访时，检查它们是否在同一个自定义网络内（`docker network inspect`）。
- 遇到连接超时，**先分清是“找不到主机”还是“端口不通”还是“服务响应慢”**。

## 5. 对待第三方服务要悲观

- 始终假设依赖的外部 API 随时可能限流、欠费、变更域名。
- 能用本地模型或内置服务的，尽量不用外部容器（**Milvus Lite > Standalone**）。
- 给 API Key 设置额度告警，避免突然欠费导致机器人哑巴。

## 6. 日志是你的眼睛

- 每次出问题，第一反应不是急着改配置，而是 `docker logs astrbot --tail 50`。
- 学会用 `grep -E "关键词"` 过滤日志，快速定位异常。
- 保留最近几天的日志，方便追溯问题发生的准确时间点。

## 7. 渐进式变更

- 一次只改一个配置，改完立刻测试。
- 不要同时调整 Prompt + 知识库 + 插件设置，出了问题无法确定根因。
- 保留旧配置的备份，出问题可以快速回滚。

## 8. 换电脑时不要慌

- 你的所有资产就是几个宿主机目录的 `.tar.gz` 文件。
- 在新电脑上安装 WSL2 + Docker Desktop，解压，重建容器，濑津美就能复活。
- 唯一可能需要重新扫码的是 NapCat。
