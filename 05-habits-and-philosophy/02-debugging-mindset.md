# 排错思维：从现象定位到根因

## 1. 排错的四项基本原则

| 原则 | 说明 |
|------|------|
| **一次只改一个变量** | 避免同时调整多个设置，否则无法归因 |
| **先重现，再修复** | 必须能稳定复现问题，才能验证修复是否有效 |
| **从底层向顶层排查** | 先确认硬件/网络/容器存活，再看配置、代码 |
| **记录每一步** | 修改了什么、观察到了什么，方便回溯 |

## 2. 排错流程（通用漏斗）
```text
问题现象
↓
确定故障范围 (是网络问题？配置问题？代码问题？)
↓
收集信息 (日志、状态、错误码)
↓
提出假设 (最可能的原因是什么？)
↓
验证假设 (修改一个变量，看现象是否变化)
↓
解决并记录 (写入笔记，避免再犯)
```

## 3. 针对容器环境的排错优先级
1. `docker ps` — 容器在跑吗？
2. `docker logs` — 日志有没有喷错误？
3. 进入容器内部测试 — `docker exec` 进去 ping/curl 目标服务
4. 检查网络拓扑 — `docker network inspect` 看看它们在不在一个网
5. 对比宿主机环境 — 在 WSL 直接 curl 测试外网、API

## 4. 常见的认知陷阱
- **归因谬误**：把同时发生的两件事认为因果（比如刚改完 Prompt 机器人就不回话，实际是 API 欠费了）。
- **忽略时间窗**：服务启动后需要初始化，比如 Milvus 启动后 1 分钟内连接失败并不代表配置错误。
- **迷信错误信息**：错误信息可能只是表象，比如 `timeout` 可能是 DNS 解析失败，而不是真的对方慢。
- **过度复杂化**：在怀疑底层网络栈之前，先检查防火墙和代理设置。

## 5. 实例复盘：长期记忆写不进去
- 现象：`/memory remember` 返回“写入失败”。
- 信息收集：日志显示 `MilvusException: server unavailable`。
- 假设1：Milvus 容器不在同一网络 → 证实，同一个 `my-bridge`。
- 假设2：Milvus 服务内部未就绪 → 查看 standalone 日志，发现 `rootcoord` 启动失败。
- 进一步收集信息：发现 WSL2 内存不足，导致 OOM。
- 最终方案：放弃 Milvus Standalone，换成 Milvus Lite（后又因依赖问题换成 APLR）。
- 反思：不要维护额外的重量级服务，能用嵌入式就不要用独立服务。

## 6. 高级排错思维
- **构建最小可复现单元**：如果复杂系统出问题，剥离一切非必要组件，只留最简配置测试。
- **二分法定位**：如果不知道是哪次修改引入的 bug，回滚到上一次确定正常的版本，逐步向前二分。
- **保留问题现场**：不要立刻修复，先用备份或快照保存问题状态供分析。
