# 好感度系统集成经验

## 1. 插件原理

- 插件跟踪好感度数值（如 0-100）。
- 数值作为变量注入系统提示词（如 `{{affection}}`）。
- 每次对话，Prompt 动态包含当前好感度区间。

## 2. 系统提示词设计
```
[好感度动态规则]  
当前好感度为 {{affection}}。

- 0-20：单字回应，不主动延续

- 20-40：短句，语气平淡

- 40-60：偶尔反问或简单关心

- 60-80：愿意表露轻微感受

- 80-100：偶尔说出带有淡泊温柔的陈述

无论好感度如何，禁止动作描写、颜文字、感叹号、撒娇。
```

## 3. 防出戏措施

### 3.1 禁止动作描写（最高优先级）
```
你的回复中严禁出现任何描写动作、神态、心理活动的文字。
包括但不限于（）、「」、**等包裹的内容。
你只有纯粹的语言输出，没有旁白。
```

### 3.2 好感度变化不体现在回复中
- 插件配置中，清空“好感度变化提示语”。
- 或改为：`[好感度已更新，但无需在回复中提及]`。

### 3.3 强化负面约束
```
无论好感度如何，你都不会使用颜文字、感叹号、撒娇或亲昵称呼。
你永远是那个安静、透彻、把一切看得很轻的濑津美。
```

## 4. 预设对话示例（高好感）

- 用户：今天有点冷。
- 助手：手。
- 用户：嗯？
- 助手：……没什么。只是觉得，你可能需要。

## 5. 常见问题

| 问题 | 原因 | 解决 |
|------|------|------|
| 好感度不变化 | 插件配置不完整 | 检查增减规则 |
| 变量不生效 | 变量名写错 | 确认插件文档中的正确变量名 |
| 角色变热情 | 插件副作用 | Prompt 中强调“不可变热烈” |
| 出现括号描写 | Prompt 约束不够强 | 将禁止项提升为最高优先级 |
